## 三、需求拆解（你现在想的 + 我帮你补全）

### 1️⃣ 城市骑行 · 绕圈 / 夜骑 / 训练

你提的点非常专业，我帮你结构化一下：

#### 🚦 路口 & 红绿灯（核心）

* 路口数量（每 km）
* 是否有信号灯
* 是否支持 **连续直行**
* 是否存在：

  * 掉头弯（U-turn）
  * 急弯（< 45°）

👉 数据来源：

* 地图 SDK 路网数据
* 路口拓扑分析（不是 LLM 的活）

---

#### 🚗 路边停车 & 干扰

* 是否为辅路 / 支路
* 是否靠近商业区
* 是否为路侧停车高发路段

👉 这一步本质是 **道路“干扰度评分”**

---

#### 🌙 夜骑专属

* 路灯密度（POI + 城市设施）
* 是否靠近：

  * 河道
  * 公园外围
  * 城市绿道
* 夜间车流量（如果能拿到历史数据，直接封神）

---

#### 🔁 绕圈训练特殊要求

* 圈长：5km / 10km / 15km
* 起点 ≈ 终点
* 尽量少打断节奏（停、等、减速）

> 这一点**现在几乎没有 App 做好**

---

### 2️⃣ 郊区 / 山区 · 小长途 & 爬坡训练

你这个方向非常对，而且是「真正骑车的人才会想的」。

#### ⛰️ 爬坡能力建模

至少三层：

* 总爬升
* 单段爬坡长度
* 平均 / 最大坡度

进阶可以做：

* 坡度分布（3–5%、6–8%、8%+）
* 是否存在「休息平台」

---

#### 🛒 补给点设计（极其重要）

* 超市 / 小卖部 / 加油站
* 间距（比如每 20–30km）
* 是否在爬坡前 / 爬坡后

👉 这一步已经是 **“路书思维”**，不是导航。

---

#### 🛑 风险 & 撤退

* 是否有：

  * 长距离无补给段
  * 无信号段
* 是否存在可提前下撤路线

## 四、技术架构（现实可落地版）

我给你一个**不玄学**的架构👇

### 1️⃣ 地图 & 路网层（基础）

* 高德 / 腾讯 / Mapbox
* 能拿到：

  * 道路类型
  * 路口结构
  * POI

这是你**硬实力下限**。

---

### 2️⃣ 路线生成（算法，不是 LLM）

核心不是一次生成一条，而是：

> **先生成 N 条候选路线 → 再评分 → 再筛选**

常见思路：

* 起点 → 目标距离 → 回环
* 起点 → 山区 POI → 返回
* 图搜索（A* / Dijkstra）+ 自定义 cost

#### 你真正要做的是：

**定义一套「骑行成本函数」**

比如：

```text
cost = 
  红绿灯 * 5 +
  掉头弯 * 10 +
  商业区 * 3 -
  绿道 * 4 -
  连续直路 * 6
```

---

### 3️⃣ LLM 真正该干的事（重点）

⚠️ 大模型 **不该** 直接算路线。

它最适合干这几件事：

#### ✅ 用户意图理解

> “今天想练 2×20 FTP，别老刹车”

→ 转成：

```json
{
  "distance": 40,
  "loop": true,
  "stop_tolerance": "low",
  "traffic_sensitivity": "high"
}
```

---

#### ✅ 路线解释 & 可读路书

* 为什么选这条
* 哪段是关键训练区
* 哪段要注意安全

---

#### ✅ 动态微调

> “这条红绿灯还是多了点”

→ 调权重，重新筛选

## 三、MVP 功能拆分（逐块）

---

## 1️⃣ 首页 / 生成页（必须极简）

**用户只需要 3 个输入：**

* 📍 **当前位置**（自动）
* 📏 **目标距离**

  * 快捷：10 / 20 / 30 / 40 km
* 🎯 **训练类型**

  * `绕圈训练（优先）`
  * （未来再加）

> MVP 里不让用户“想太多”

---

## 2️⃣ 意图解析（LLM · 超轻量）

### 输入（示例）

```text
我想在家附近绕圈骑 30 公里，别老停
```

### 输出（结构化）

```json
{
  "mode": "urban_loop",
  "distance_km": 30,
  "stop_tolerance": "very_low",
  "u_turn_allowed": false,
  "priority": ["continuity", "safety"]
}
```

👉 这一步的价值：

* **兜底自然语言**
* 给非参数党一个入口

---

## 3️⃣ 路线候选生成（MVP 最硬核部分）

### 策略（不要一步到位）

* 从当前位置向 4–8 个方向发散
* 构造闭环
* 每条：

  * 距离误差 ±10%
  * 不穿主干高架
  * 不进小区内部路

> 先生成 **10–20 条候选**

---

## 4️⃣ 路线评分模型（MVP 版）

这是你 MVP 的“灵魂”，但可以**先很粗糙**。

### 核心指标（必做）

| 指标    | 说明        |
| ----- | --------- |
| 红绿灯数  | 路口信号灯     |
| 掉头弯   | U-turn    |
| 急弯    | < 45°     |
| 路段连续性 | ≥ 500m 直行 |
| 路线闭合度 | 起终点距离     |

### 简化评分公式

```text
score =
  - 红绿灯 * 5
  - 掉头弯 * 10
  - 急弯 * 4
  + 连续路段 * 3
```

👉 MVP 只要能 **明显比随机路线强** 就够。

---

## 5️⃣ 路书生成（LLM 真正发光）

### 输出示例（给人看的）

> 这是一条 **约 6km 的绕圈路线**，全程 4 个红绿灯，没有掉头弯。
> 第 2–5km 是连续直路，适合做稳定功率输出。
> 夜骑建议注意第 1km 路段照明稍弱。

### 同时生成结构化路书（给机器）

```json
{
  "loops": 5,
  "key_segments": [
    { "start": 2.1, "end": 5.0, "type": "steady_effort" }
  ],
  "warnings": ["low_light"]
}
```

---

## 6️⃣ 地图展示 & 导出（闭环）

* 地图显示路线

## ✅ 什么是高德 MCP Server？

**高德地图 MCP Server** 是基于 **Model Context Protocol (MCP)** 封装的一套地图工具集合，核心作用是：

👉 **让大模型（LLM）可以直接调用地图能力，无须自己写各种 API 请求逻辑**，它内部封装了高德丰富的地理位置服务。 ([腾讯云开发][1])

它提供的能力大致包括：

| 路线规划（包括骑行）  | ✅    |              |

| 天气查询        | ✅    |              |
| 坐标转换        | ✅    | ([腾讯云开发][1]) |

## 📌 它对你这个项目 MVP 最核心的三类能力

### ✅ 1）骑行路线规划能力（maps_bicycling）

这是最关键的功能之一：

* MCP Server 内置了骑行路线规划接口，可以生成从 A → B 的推荐骑行路线。
* 支持路线规划距离可达 **500 km**，足够应对城市绕圈和郊区长距离路线。 ([腾讯云开发 CloudBase][2])

  * 多点→单点的距离矩阵
  * 更复杂的路段组合评分（你未来可能需要）

这对你的路线筛选算法和“连续性”、“绕圈性”等指标都很有用。 ([腾讯云开发 CloudBase][2])

## ⚠️ 需要理解的两点限制

### ⚠️ MCP Server 不会替代你所有逻辑

它能提供路径规划（单一 A→B）和 POI 数据，但它不会自动：

❌ 生成**回环路线**
❌ 围绕起点做“连续循环 + 少红绿灯”
❌ 自动计算爬坡等级 / 累计爬升
❌ 预先过滤掉“急转弯 / U-turn”

这些 **都还是需要你自己在业务层面做算法处理和二次筛选**。

---

### ⚠️ MCP 是一种辅助协议，而不是地图 SDK 取代

它是一种 **让 LLM 调用高德 API 的统一规范**，它本身只是把现有 API 整合成对 AI 更友好的方式。它不会给你内置的“训练友好路线”模式。 ([Reddit][4])
